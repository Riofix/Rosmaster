# STM32 底盘运动控制固件 (STM32F103)

该固件主要负责上位机（如 RDK X5 / ROS / PC）与下位机 Emm_V5 步进闭环电机驱动器之间的数据桥接、协议解析与调度控制，使用基于无锁环形队列和表驱动分发的架构。

## 通信架构流向
上位机指令 ➔ **(USART2 RX/DMA)** ➔ **协议解析状态机 (Protocol)** ➔ **无锁环形队列 (MsgQueue)** ➔ **主轮询分发引擎 (App_Protocol_Tick)** ➔ **建表执行 (Emm_V5)** ➔ 串口流向下位电机。

## 串口通信协议格式

固件与上位机之间的通信严格遵守以下帧格式：

| 帧头 (Header) | 命令字 (CMD) | 长度 (LEN) | 数据 (DATA) | 校验和 (CHECKSUM) |
|:---:|:---:|:---:|:---:|:---:|
| 1 Byte | 1 Byte | 1 Byte | N Byte (Max 32) | 1 Byte |

*   **接收包头**：`0xFF` (用于 STM32 接收上位机控制)
*   **发送包头**：`0xFB` (用于 STM32 向上位机反馈)
*   **校验和**：`Header + CMD + LEN + DATA[0]...DATA[N]` 的累加和（`uint8_t`，溢出截断）。

## 协议指令映射表

### ⬇️ RX 接收部分 (上位机 -> STM32：0x50 ~ 0x5C)

下发的控制字与 Emm_V5 的 13 条指令严格对应，包含 **12条控制配置指令** 与 **1条查询指令**。
默认情况下 `DATA` 的第0个字节 (`DATA[0]`) 均为目标电机地址（`addr`）。

| CMD (十六进制) | 指令功能描述 | 对应 Emm 驱动动作 |
| :---: | :--- | :--- |
| **0x50** | 脱机与使能控制 | `Handle_En_Control` |
| **0x51** | 速度模式控制 | `Handle_Vel_Control` |
| **0x52** | 位置模式控制 | `Handle_Pos_Control` |
| **0x53** | 紧急停止运行 | `Handle_Stop_Now` |
| **0x54** | 触发多机同步开始运动 | `Handle_Sync_Motion` |
| **0x55** | 设置单圈绝对回零位置 | `Handle_Origin_Set_O` |
| **0x56** | 修改回零相关的运动参数 | `Handle_Origin_Modify_Params` |
| **0x57** | 触发回零动作 | `Handle_Origin_Trigger` |
| **0x58** | 强制打断并终止回零 | `Handle_Origin_Interrupt` |
| **0x59** | 修改控制模式 (开环/闭环) | `Handle_Modify_Ctrl_Mode` |
| **0x5A** | 重置当前实时位置为 0 | `Handle_Reset_Cur_Pos` |
| **0x5B** | 强制解除电机的堵转保护 | `Handle_Reset_Clog_Pro` |
| **0x5C** | **读取指定的系统参数/状态** | `Handle_Read_Sys_Params` (查询) |

### ⬆️ TX 发送部分 (STM32 -> 上位机：0x60 ~ 0x63)

当底板解析通过并尝试驱动电机后，或遇到自身错误时，向上位机反馈的结果指令：

| CMD (十六进制) | 指令功能描述 | 详细说明与用途 |
| :---: | :--- | :--- |
| **0x60** | `CMD_TX_ACK_PARAM` | **状态查询返回**。专门作为对 `0x5C` 查询指令的应答，包含电流、位置、状态等电机上报参数的真实数值。 |
| **0x61** | `CMD_TX_ACK_OK` | **通用握手成功应答**。当收到指令检查无误并送入底座执行后，通常反馈一次 OK 代表指令顺畅通达。 |
| **0x62** | `CMD_TX_ACK_ERR` | **异常报警应答**。指令错误或非法负载导致系统拒收时，会携带 `ERR_CODE` 报错字弹回该指令给上位机报警。 |
| **0x63** | `CMD_TX_REPORT_POS` | **位置到达主动上报**。设计预留以用作当底层电机走到靶向目标位后，无需上位机轮询而主动上报“已就位”的信号提醒。 |

## 工程架构特性

1.  **无锁并发安全设计**：底层硬件中断产生数据流（单生产者），App层以业务轮询消费（单消费者），无需 FreeRTOS，不关中断也绝生死锁。
2.  **死锁断帧自首系统**：发生串口数据掉帧破损时，超时器和满队抛弃机制会强制打落错位数据。
3.  **消除庞大`switch`分支**：以静态函数回调挂载于一维数组表，查找效率高达 `O(N)`~`O(1)`，随时支持插拔增加任何新增控制逻辑。
