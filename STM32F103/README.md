# STM32F103 机器人/控制主板工程

## 1. 项目简介
这是一个基于 STM32F103 微控制器的嵌入式控制项目，主要用于构建具备无线远程通信与高精度电机控制的设备（如机器人底盘、云台、机械臂等）。项目利用了 ESP8266 实现 WiFi 透传通信，能够有效解析一套具备容错校验的自定义二进制通信协议，并基于该架构驱动 EMM_V5 步进/伺服智能电机控制器，同时支持利用 OLED 屏幕直观展示系统状态及受控信息。

## 2. 硬件架构
*   **主控 MCU**：STM32F103 (常见的如 C8T6 核心板)
*   **无线通信网络**：ESP8266 Wi-Fi 模块（工作于透传 TCP Client 模式，经由 USART + DMA 交互）
*   **人机交互节点**：I2C OLED 屏幕（128x64常见规格，用于呈现初始化进程、网络连接状态及收报日志）
*   **电机驱动模组**：EMM_V5 闭环步进或伺服控制器（借由 USART 回接协议解析）

## 3. 核心功能模块指南 (Src/ 下源文件说明)
*   **系统与外设基础驱动层 (BSP)**
    *   `bsp_systick.c / .h`: 提供精确的毫秒级与微秒级阻塞延时功能。
    *   `bsp_usart.c / .h`: 核心串口通信层。采用了软硬结合的 **DMA 配合环形缓冲区（Ring Buffer）** 架构，从根本上防止了大量传感器指令并发时产生的数据丢失。
    *   `bsp_iic.c / .h`: 软件模拟 I2C 总线，为了规避部分 STM32 硬件 I2C 锁死 bug 而设，主要服务于显示模块。
*   **外设中间层扩展应用 (BSP)**
    *   `OLED.c / .h`: 提供了点阵绘图、格式化英文字符串写入、清屏与指定行覆盖等 OLED 图形 API。
    *   `bsp_esp8266.c / .h`: 打包了极为关键的 ESP8266 AT 指令序列。单片机上电会将其硬复位，随后进行联网认证并开启透传。
*   **协议栈与业务解析层 (APP/BSP)**
    *   `protocol.c / .h`: 项目专属的核心通讯中间件。包含一个完全解耦的**状态机包解析器**和**快速封包发送器**。它独立运作于 `rx_buf` 之上，自动剥离无用干扰字节。
    *   `app_protocol.c / .h`: 面向应用层的具体业务实现。负责接管解析好的数据包，基于首字节的 `Motor Address` 定向分发电机控制命令，并利用 Flash 实现本底参数固化。
    *   `protocol_emm.c / .h` 及 `Emm_V5.c`: 对 EMM_V5 智能电机驱动器特定通讯报文标准进行再次封装的对接层。

## 4. 软件串口通讯协议与指令字典
项目中运行了严格的数据包检验机制。分为 **接收（RDK -> STM32）** 与 **发送（STM32 -> RDK）** 两条流，二者采用**不同的帧头**。

### 4.1 数据帧基础结构
| 节点标识 | 帧头 (1 Byte) | STM32级总线ID (1 Byte) | 命令字CMD (1 Byte) | 数据长度Len (1 Byte) | 数据体Data (N Bytes) | 校验和 (1 Byte) |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: |
| **主机下发** | `0xFF` | `0x0A` (默认) | 指令功能 | `N` (范围0~255) | 第一字节必为电机Addr | `Sum(Header...Data) & 0xFF` |
| **从机上传** | `0xFB` | `0x0A` | 成功/报错/状态 | `N` (范围0~255) | 详见下方定义 | `Sum(Header...Data) & 0xFF` |

> *注：协议帧中的 ID 字段代表的是**这块 STM32 开发板在整台 RDK 机器人系统里的节点 ID**。由于 STM32 内部会穿透挂载多个独立电机，因此在操作电机时，具体的电机地址(Addr) 存放在 Data 域的首字节。*

### 4.2 具体业务控制指令表 (CMD 词典)

#### RDK X5 -> STM32 从机（接收下发操作，0x50 ~ 0x59）
| 宏定义名称 | 命令字 | Data长度 | Data 域物理结构 (RDK X5发送) |
| :--- | :---: | :---: | :--- |
| `CMD_RX_READ_PARAM` | `0x50` | 2 Byte | **精细参数查询**：获取任意电机指定状态<br>`[0] 电机地址(addr)`<br>`[1] 参数枚举(SysParams_t)` (如:8=目标位置, 9=实时转速, 10=实时位置, 15=系统状态) |
| `CMD_RX_CTRL_SPEED` | `0x51` | 5 Byte | **速度模式** (连续旋转)：<br>`[0] 电机地址`<br>`[1] 方向`: (0=CW反转, 1=CCW正转)<br>`[2-3] 目标速度(RPM)`: uint16_t (高位先收待配)<br>`[4] 加速度`: uint8_t |
| `CMD_RX_CTRL_POS` | `0x52` | 10 Byte | **位置模式** (点位移动)：<br>`[0] 电机地址`<br>`[1] 方向`<br>`[2-3] 运行速度(RPM)`: uint16_t<br>`[4] 加速度`<br>`[5-8] 移动脉冲数(clk)`: uint32_t<br>`[9] 相对/绝对`: (0=相对增量, 1=绝对坐标) |
| `CMD_RX_STOP` | `0x53` | 2 Byte | **急停 / 清除堵转**：<br>`[0] 电机地址`<br>`[1] 释放动作`: 0=仅急停停止输出, 1=解除堵转保护锁定 |
| `CMD_RX_ENABLE` | `0x54` | 2 Byte | **电机脱机/使能控制**：<br>`[0] 电机地址`<br>`[1] 状态`: (0=释放/脱机手盘, 1=锁轴/使能) |
| `CMD_RX_ORIGIN` | `0x55` | 2 Byte | **触发回零**：<br>`[0] 电机地址`<br>`[1] 寻零模式` |
| `CMD_RX_SYNC_RUN` | `0x56` | 1 Byte | **多机同步触发**：针对预先下发的缓冲同步指令，触发指定运动。<br>`[0] 电机地址` (设为 0x00 代表广播所有电机同步起步) |
| `CMD_RX_SET_ID` | `0x59` | 1 Byte | **更改STM32开发板ID**：此命令直达STM32。<br>`[0] 新的节点ID(1~255)`。立即写入 Flash 生效（重启不丢）。 |

#### STM32 从机 -> RDK X5（发送应答状态，0x60 ~ 0x69）
| 宏定义名称 | 命令字 | Data长度 | Data 域物理结构 (STM32上传) |
| :--- | :---: | :---: | :--- |
| `CMD_TX_ACK_PARAM` | `0x60` | 6 Byte | **参数查询返回(应答 0x50)**：<br>`[0] 电机地址`<br>`[1] 被询参数枚举(SysParams_t)`<br>`[2-5] 参数结果值`: 固定统一使用 4 Byte 回传，RDK上位机按需强转读取。 |
| `CMD_TX_ACK_OK` | `0x61` | 2 Byte | **通用握手/成功心跳**：<br>`[0] 电机地址(addr)`<br>`[1] STM32成功接收执行的源指令字(CMD)` |
| `CMD_TX_ACK_ERR` | `0x62` | 3 Byte | **异常报警与拒接**：<br>`[0] 电机地址(addr)`<br>`[1] 失败的指令字`<br>`[2] 错误码` (0x01:Flash读写失败, 0x02:参数不合规, 0x03:总线失联命令不存在等) |
| `CMD_TX_REPORT_POS` | `0x63` | 5 Byte | **位置达标主动上报**：开启定距位移后，电机走完即刻触发上报：<br>`[0] 电机地址(addr)`<br>`[1-4] 停下的绝对脉冲位置` (uint32_t) |


## 5. 开发调试与使用指南
1.  **开发IDE**：双击根目录下的 `Slave.uvprojx` 即可在 **Keil MDK-ARM**（推荐 v5 以上版本）中无缝启动工程。目前请将 `app_protocol.c` 手动拖入 Keil Source Group 内参与编译。
2.  **上电运行流转表现**：
    *   单片机接电启动会在 OLED 首行显示 `"Init..."` 进行模组自检。
    *   紧接着进入 WiFi 连接阻塞阶段（OLED 第二行亮起 `"WiFi Connecting"`）。若此时超时连不上，会停在此处不断重试。
    *   TCP 网络挂载成功后，OLED 清屏并呈现 `"Waiting Data:"`，此时进入核心死循环监听 DMA 协议报文流。协议初始化会自动挂载读取位于 Flash(`0x0800FC00`) 处的板级节点ID。
3.  **避坑指南**：
    *   调试时如果出现上电**完全黑屏并且单片机疑似死机**，极大可能是由于 **I2C引脚接错（SCL与SDA反相）** 或者忘记接加上拉，导致 I2C 模拟时序发生电平阻塞死锁，交换线序或补接拉高器重启即可。

## 6. 代码扩展与维护
- 如果你希望研发新的上层动作控制流（如：点亮LED板灯或查询传感器流编解码），你应该在 `app_protocol.h` 中新设一组 `#define CMD_XXX` 并随后直接进入 `app_protocol.c` 内 `App_Protocol_Handler` 对应的 `switch(cmd)` 分支中堆叠执行分支。
- 需要将这套代码向非 F1 系列的板卡（如 STM32F4/H7）移植时，请首先修改 `stm32f10x_it.c` 内的中断入口服务以及 `bsp_usart.c` 针对不同 HAL/StdPeriph 库的外设初始化代码，**整套通讯协议层（含app路由层）则是完全一字不差地无缝跨平台代码，无须更改**。

## 7. 版本更新日志
* **v1.1.0** (Current)
  - **横向扩展**：抽象业务层至专属的 `app_protocol`，彻底解耦底层协议帧解析与控制流逻辑。
  - **中间件多极路由**：支持由 RDKX5 经由节点 STM32 定向操作总线下属的多台 Emm_V5 从电机的地址（`addr`）软路由下发机制。
  - **系统级定址持久化**：全天候使用内部 Flash 末页 (`0x0800FC00`) 对 STM32 本机主节点 ID 进行防断电记忆存储保护。
  - **收发防扰隔离**：接收包头(`0xFF`)与发送防干扰包头(`0xFB`)实现通道互锁。
